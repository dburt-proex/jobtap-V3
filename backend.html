Operations backend

Airtable stays the operator cockpit.

JobTap uses Airtable for early visibility, admin review, and routing management. It should not become the long-term production database.

Structured operations model

From public intake to reviewed dispatch.

Website submissions should flow through a secure API endpoint, then stage records in Airtable for review, matching, tasks, and follow-up.

Website Intake QueuePublic submissions staged for review and promotion.
JobsService request, urgency, value, score, and routing status.
ContractorsCoverage, trade type, qualification, and availability.
MatchesRanked contractor options with score and dispatch order.
TasksOperator follow-up created by routing lane and policy.
Routing RulesEditable weights for urgency, service fit, and score lanes.

Production path

Airtable is correct for early operations. The production system should eventually move writes behind an API and store core records in Postgres or a similar production database.

NowStatic site plus Airtable-ready payloads.
NextSecure API endpoint writes validated intake records.
ScaleProduction database becomes system of record.
AdminAirtable remains useful as a cockpit/reporting layer.
